# Process Assets 项目目标

## 总目标

用 GitHub 建立一个可以展示流程数字化、低代码业务分析和运营系统设计能力的个人作品资产库。

这个仓库不是单纯为了“学代码”，也不是零散存放文档，而是把过去在行政管理、人事管理、费控报销、内部流程梳理、制度标准化、低代码工具搭建中的经验，沉淀成可以展示、复盘和复用的数字资产。

仓库最终要体现三类能力：

- 从业务问题中识别流程节点、责任边界和风险点
- 把线下经验转成表单、状态、审批、规则和数据结构
- 用案例、方法和模板证明自己能设计可落地的运营系统

---

## 当前仓库定位

仓库采用“专题 + 案例 + 方法 + 模板”的结构。

```text
process-assets
-> expense/
   -> cases/
   -> methods/
   -> templates/
-> HR/
   -> cases/
   -> methods/
   -> templates/
-> GOAL.md
-> README.md
```

每个专题都不是单个流程说明，而是一组可持续扩展的流程数字化资产。

## 已进入建设的专题

### 1. 费用管理专题 `expense/`

费用专题用于展示企业费用流程、项目费用管理、预算前置、过程控制、事后复盘和风险校验能力。

当前重点内容：

- 非项目制通用经营支出管理

- 费用分类方法
- 流程问题识别方法
- 费用检查和案例沉淀模板

费用专题已经形成较完整的案例、方法和模板结构。

### 2. 人力资源专题 `HR/`

HR 专题用于展示从用人需求、招聘、人效评估、人才库、入职、员工生命周期、薪酬福利绩效到数据报告的完整人力资源运营系统设计能力。

当前规划内容：

- HR Operating System：完整 HR 运营系统案例
- Employee Lifecycle：入转调离与人事生命周期案例
- Payroll and Cost：薪酬核算与人工成本案例
- HR 节点识别方法
- 人事档案管理方法
- 薪酬成本核对方法
- 招聘需求、入职清单、薪酬检查模板

HR 专题是当前下一阶段的重点建设方向。

## 案例统一写法

每个案例尽量保持统一结构，方便阅读、复用和对比。

```md
# 案例名称

## 背景

## 原流程痛点

## 标准化后的流程

## 表单字段设计

## 审批节点设计

## 状态流转

## 异常场景处理

## 可数字化部分

## 需要人工判断的部分

## 数据输出

## 复盘
```

不同专题可以按业务需要微调，但都要体现：

- 业务场景是什么
- 原来哪里乱、慢、断、不可追踪
- 标准化后怎么流转
- 系统里需要哪些字段和状态
- 哪些节点需要审批或人工判断
- 最终能沉淀什么数据

## 方法与模板目标

案例用于展示“做过什么、怎么拆”。

方法用于沉淀“以后遇到类似流程怎么判断”。

模板用于沉淀“下次可以直接复用什么”。

当前方法论方向：

```text
流程收集
-> 问题识别
-> 节点拆解
-> 字段设计
-> 状态流转
-> 审批规则
-> 风险控制
-> 数据输出
-> 案例复盘
```

## 阶段计划

### 第一阶段：稳定作品集结构

- 保持根目录 README 能说明仓库定位和阅读顺序
- 保持每个专题都有 README、cases、methods、templates
- 保持案例命名和文档结构尽量统一

### 第二阶段：完成两个核心专题

- 继续完善 `expense/`，让费用专题成为成熟样板
- 重点补齐 `HR/`，让 HR 专题从空目录变成完整案例组

### 第三阶段：扩展更多运营流程专题

后续可以按经验逐步扩展：

- 行政采购
- 合同与用章
- 固定资产
- 内部服务工单
- 供应商管理
- 组织效率分析

### 第四阶段：加入轻量技术表达

围绕业务场景学习和补充：

- Markdown：写案例、模板、方法
- Git/GitHub：版本管理和作品展示
- JSON/YAML：表达字段、规则和配置
- HTML/CSS：制作简单展示页
- Python 或 JavaScript：制作流程检查小工具

学习路径：

```text
Markdown 文档资产
-> GitHub 作品集
-> 表单字段配置 JSON
-> 简单网页展示
-> 简单流程检查工具
```
